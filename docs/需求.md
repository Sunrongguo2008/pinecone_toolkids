# 角色与任务
你是一个资深的 Windows 桌面应用开发专家，精通 C# 与 WinForms，熟悉 Windows 底层 API（注册表、进程/作业对象、WinPE 环境）。
现在需要你从零开始编写一个“便携式工具箱”软件（类似图吧工具箱），核心特色是：
1. 软件分类管理；
2. 软件“便携化”运行（启动前还原、退出后备份并清理，做到用完不留配置垃圾——本质是**备份/还原**，不是运行期真隔离）；
3. 轻量级的沙盒扫描（快照对比，辅助生成配置）。

**第一优先级是兼容性**：必须能在 Windows 7 SP1 ~ Windows 11，以及 WinPE 下运行。为兼容性，**不追求 Win11 视觉效果（云母/亚克力/圆角），采用中规中矩的扁平外观即可**。

# 技术栈（已定）
- 运行时：**.NET 6**（`net6.0-windows`）。原因：它是**最后一个官方支持 Windows 7 SP1 的 .NET**，且能发布为**自包含单文件**，满足“免装运行时 + 进 PE”的要求。（.NET 7/8 已不支持 Win7，排除；.NET Framework 4.8 不能自包含、PE 需集成 NetFx，与“自包含”目标冲突，排除。）
- UI 框架：**WinForms**。原因：用 GDI 绘制，**不依赖 DWM/Direct3D**，是 WinPE 里最稳的 GUI；体积小；“中规中矩”的外观正合需求。（WPF 在 PE 只能软件渲染、有起不来的风险，故不用。）
- 发布方式：**自包含 + 单文件**（`--self-contained true -p:PublishSingleFile=true`）。
- 目标位数：**仅 x64**（覆盖 Win7 x64 / Win8 / Win10 / Win11 以及 x64 PE）。32 位老机器 / 32 位 PE 暂不支持，如后续需要再补 x86。
  - 注意：64 位 WinPE 默认常不带 WoW64，无法运行 32 位程序——这也是只发 x64 的原因之一。
  - 工具箱自身位数不影响沙盒：注册表的 32/64 视图（`RegistryView.Registry32/64`）在代码里都显式处理。
- 配置格式：JSON。
- 第三方依赖（尽量少，避免造轮子）：
  - **Newtonsoft.Json**（容错解析用户手写配置）——唯一硬依赖。
  - 注册表：系统自带 `Microsoft.Win32.Registry` + 调用 `reg.exe import/export`。
  - 图标提取：自带 `System.Drawing` / Win32 `SHGetFileInfo`。
  - 等待多进程程序退出：Win32 **Job Object**（P/Invoke）。
  - UI 主题（可选）：Krypton Toolkit 或 ReaLTaiizor（纯托管，做深色扁平主题）；不引入也可用原生控件。
- 权限：通过 `app.manifest` 设置 `requireAdministrator`；同时声明 DPI 感知、`longPathAware`，以及 Win7~Win11 的 `supportedOS` GUID。

# 兼容性要求
| 环境 | 是否支持 | 说明 |
|---|---|---|
| Windows 11 | ✅ | 自包含直接跑 |
| Windows 10 | ✅ | 自包含直接跑 |
| Windows 8/8.1 | ✅ | 自包含直接跑 |
| Windows 7 SP1（x64）| ✅ | 裸机可能需先打 **SHA-2 补丁 KB4474419**；少数情况需 VC++ 运行库 |
| WinPE（x64） | ✅(需验证) | 自包含**免装 .NET**；UI 走软件渲染。**正式开发前先用最小程序在目标 PE 验证能否弹窗** |
| WinPE（x86） | ❌ | 仅发 x64，不支持 32 位 PE |

PE 相关注意：
- PE 系统盘多为只读/临时（X: 内存盘）。工具箱的配置与备份目录必须**可配置**，且当所在目录不可写时**自动回退**到可写位置（如 U 盘其它目录、或 `%TEMP%`）。
- PE 的注册表是易失的，沙盒的“备份/清理”在 PE 下意义不大：程序应**自动检测 PE**（判断 `HKLM\SYSTEM\CurrentControlSet\Control\MiniNT` 是否存在），可按设置在 PE 下跳过备份/清理，只负责“把工具启动起来”。

**PE 实测结论（2026-06，已通过）**：在基于 Win11（10.0.26200）的 x64 WinPE 中，单文件自包含版可正常启动并弹窗。
- PE 检测：`MiniNT` 法准确（佐证：机器名 `MININT-*`、用户 `WindowsPE$`、系统盘 `X:`）。
- 程序目录（实测从 D: 运行）与 `%TEMP%` 均可写；但 `%AppData%`/`%LocalAppData%` 解析到 `X:\Users\Default\...`（内存盘，重启即失）——印证沙盒在 PE 下意义不大，应默认 `skipSandboxInPE`，配置/备份优先写程序目录、不可写时回退 `%TEMP%`。
- Win7 x64 尚未实测（风险较低，可用同一个 PeTest.exe 随时验证；必要时打 KB4474419）。

# 核心功能需求

## 1. 界面与交互设计
- **整体风格**：扁平、规整、简洁，**深色为主**（亦可切浅色）；不使用亚克力/云母/圆角等需 DWM 的效果，确保 Win7/PE 下表现一致。
- **主界面布局**：左侧为分类导航栏，右侧为内容区。点击左侧分类，右侧以**大图标网格**（或列表）显示该分类下的软件图标与名称。布局（grid/list）可在配置中切换。
- **设置页面**：允许手动添加、删除、修改分类，以及向分类中添加、删除软件；可设置主题、数据目录、是否在 PE 跳过沙盒等。
- **软件右键菜单/按钮**：包含“运行”、“编辑配置”、“扫描沙盒”、“删除”等操作。
- 运行软件期间界面不卡死（异步等待），并显示“运行中”状态。

## 2. 目录结构与配置文件
```
工具箱根目录/
├─ Toolkids.exe                 主程序（自包含单文件）
├─ kidconfig.conf               全局配置（后缀 .conf，内容为 JSON）
└─ Data/                        所有软件的容器目录（可在配置里改名/改路径）
   └─ CPU-Z/                    每个软件一个独立子文件夹
      ├─ toolconfig.json        该软件的启动信息 + 沙盒规则
      ├─ Apps/                  存放该软件的实际程序文件
      └─ Backup/               （按需自动创建）备份的注册表(.reg)与目录
```
> 说明：顶层容器固定叫 `Data`；每个软件文件夹内部用 `Apps/` 放程序、`Backup/` 放备份。（修正了原文档里“软件目录(Apps)”与“Data”混用的歧义。）

### 全局配置 `kidconfig.conf`（JSON）
```json
{
  "version": 1,
  "theme": "dark",                 // dark | light
  "layout": "grid",                // grid | list
  "dataDir": "Data",               // 相对根目录；不可写时自动回退
  "categories": [
    {
      "id": "cpu",
      "name": "处理器工具",
      "apps": ["CPU-Z", "CoreTemp"] // Data 下的子文件夹名
    }
  ],
  "settings": {
    "confirmConflictBeforeRestore": true, // 还原前若系统已存在同名项，弹出可滚动列表确认
    "askBackupOnExit": true,              // 退出后询问是否备份+清理
    "skipSandboxInPE": true,              // 检测到 PE 时跳过备份/清理
    "snapshot": {                          // 沙盒扫描的默认范围
      "registryRoots": ["HKCU\\Software", "HKLM\\Software"],
      "fileRoots": ["%AppData%", "%LocalAppData%", "%ProgramData%"]
    }
  }
}
```

### 软件配置 `toolconfig.json`（每个软件一个）
```json
{
  "name": "CPU-Z",
  "description": "CPU/主板检测工具",
  "icon": "Apps\\cpuz_x64.exe",     // 从该 exe 提取图标；也可填 .ico/.png 相对路径
  "launch": {
    "exe": "Apps\\cpuz_x64.exe",    // 相对“本软件文件夹”
    "args": "",
    "workingDir": "Apps",
    "runAsAdmin": true,
    "waitMode": "tree"              // none | process | tree | named
    // "waitProcessName": "cpuz"    // waitMode = named 时指定要等待的进程名
  },
  "sandboxRules": {
    "registry": ["HKCU\\Software\\CPUID\\CPU-Z"],
    "files": ["%AppData%\\CPUID\\CPU-Z"]   // 支持 %AppData% %LocalAppData% %ProgramData% %UserProfile% %Temp% 等变量
  }
}
```

## 3. 便携化（备份/还原）运行逻辑
点击“运行”一个配置了 `sandboxRules` 的软件时：
1. **启动前（还原）**：若该软件目录下有 `Backup`，将其中备份的注册表项(.reg)与目录复制/导入到系统对应位置。
2. **冲突处理**：写入系统前，若发现系统已存在同名注册表项或文件夹，弹出提示框，内含**可滚动文本框**详列将被覆盖的内容。用户可选“覆盖”、“取消”，或**勾选部分项**进行部分覆盖。
3. **启动软件**：按 `launch` 配置（必要时以管理员权限）启动目标程序，并按 `waitMode` 等待其退出：
   - `process` 仅等被启动的进程；`tree` 用 **Job Object** 等待整棵进程树（默认，解决启动器秒退问题）；`named` 等待指定进程名结束；`none` 不等待。
4. **退出后（备份与清理）**：软件退出后，按设置弹窗询问是否将 `sandboxRules` 指定的注册表与目录从系统导出/复制回 `Backup`，随后从系统删除这些项，实现无痕化。
   - 删除时若被占用（句柄未释放），需**重试/延迟**。
   - 注册表读写需按工具箱与目标程序的位数处理 **32/64 视图**。

健壮性要求：
- 整个还原/备份/清理过程要有**事务日志(journal)**，中途崩溃/断电后，下次启动能检测到未完成操作并提示续做或回滚，避免丢备份或留残留。
- 在 PE 且 `skipSandboxInPE = true` 时，跳过 1/2/4 步，仅执行启动。

## 4. 沙盒扫描功能（轻量级快照法）
右键“扫描沙盒”，辅助生成/更新 `toolconfig.json`。
- **原理（快照对比）**：启动软件前对**限定范围**的注册表与文件目录拍快照，软件关闭后再次拍快照并对比，算出新增/变更的注册表项与文件。
  - 范围由 `kidconfig.conf` 的 `settings.snapshot` 控制（默认仅扫 `HKCU\Software`、`HKLM\Software` 及 `%AppData%`/`%LocalAppData%`/`%ProgramData%`），**不做全注册表/全盘扫描**（不可行）。
- **结果应用**：扫描完成后弹窗展示差异列表供勾选；点击“一键写入配置”，把选中项写入该软件 `toolconfig.json` 的 `sandboxRules`（路径自动还原为 `%AppData%` 等变量形式）。

## 5. 健壮性与日志
- 写一个轻量文件日志（记录启动、还原、备份、清理、错误）。
- 所有破坏性操作（写注册表、删目录）前有确认或日志留痕。
- 出错时给用户可读的中文提示，而不是直接崩溃。
